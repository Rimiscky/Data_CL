Traitement

Pipeline ETL — Étapes détaillées

1

Ingestion

Appel API avec retry 3× et backoff exponentiel. Sauvegarde JSON + CSV horodatés dans data/raw/.

2

Extraction

Lecture du fichier JSON le plus récent par région. Conversion en DataFrame pandas.

3

Transformation

Renommage colonnes, conversion UTC, remplissage NaN=0, enrichissement temporel (heure, weekend, trimestre), métriques dérivées.

4

Fusion Énergie × Météo

merge_asof sur datetime, tolérance 1h. Ajout catégories : temp_category, wind_category, is_rainy. Taux de correspondance : 100%.

5

Chargement

Sauvegarde latest.csv + partitions year=YYYY/data_ts.csv pour chacune des 4 régions.

Choix d'ingénierie

Décisions techniques clés

🔀
ETLmerge_asof au lieu d'un join classique

Les timestamps énergie (demi-horaire) et météo (horaire) ne sont jamais parfaitement alignés. merge_asof avec tolérance 1h permet une jointure temporelle approximative sans perdre de lignes. Résultat : 100% de correspondances.

📦
DashboardDonnées embarquées JSON dans le HTML

Les dashboards HTML sont des fichiers autonomes (pas de backend). Les données (500 lignes) sont injectées dans window.D à la génération. Avantage : zéro dépendance serveur, cacheable, distribuable.

🔒
SecretsEnvironmentFile systemd au lieu de .bashrc

.bashrc n'est pas sourcé par les services systemd. Solution : EnvironmentFile=/home/ubuntu/.env dans l'unité systemd. Clé RTE disponible dans le service sans l'exposer dans le code.

📌
FrontendCDN Plotly fixé à 2.27.0

plotly-latest.min.js peut rompre les rendus à chaque mise à jour. Version fixée : comportement déterministe en production, même hors ligne si le CDN est down.
